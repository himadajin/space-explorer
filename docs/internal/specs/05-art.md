# 50 — art: 絵作りの原理と調整定数

この文書の内容はすべて表示専用であり、identity に影響しない。原理は [方式]、数値は [調整]。参照イメージは深宇宙写真(Hubble Deep Field)、宇宙論シミュレーションの可視化、Gaia 全天マップであり、SF・ゲームの画面設計ではない。

## 原理

1. 二層の光。すべての光点は「2〜3px の鋭い芯 + 極めて淡く広いハロ」で描く。鮮鋭さは芯が、壮大さは暗闇の中の無数の微光が担う。単層の柔らかいグラデーションはピンボケ(玉ボケ)に見えるため禁止。
2. 無彩色基調。色は象牙色〜淡青の狭い色温度レンジに収める。種類の区別は色相ではなく明るさと形で伝える。彩度の高い記号色(ゲーム UI 的な橙・水色・紫)は使わない。表示用の脱色は desat(css, t)(白方向への線形補間)で行い、安定パラメータには触れない。
3. 輝度のべき乗則。大多数は暗く、明るい天体はごく少数。一様な明るさの点の集まりは人工物に見える。
4. 線と図解の禁止。フィラメントは線分として描かない(密度だけが構造を語る)。軌道線は描かない。構造は点の密度と輝度の分布から浮かび上がらせる。
5. 観測者は媒質の内側。宇宙網の描画データの果てはフェザリング(30-cosmic-web.md)で闇に溶かし、輪郭として知覚させない。ズーム上限(40-zoom.md)で外側からの俯瞰を禁じる。
6. 統計的な滲みは物体ではない。集合点群の拡大禁止と溶解(40-zoom.md の entered 規律)。
7. 露出の規律。ACES トーンマッピング + 控えめな露出で加算合成の白飛びを抑える。画面平均輝度は低いほど、まばらな光が壮大に見える。

## テクスチャ生成(手続き、決定論)

すべてのテクスチャは canvas 上で、texSeed = parseInt(fingerprint, 16) を種とする mulberry32 から生成される。つまり見た目は安定パラメータのみの関数である。

- 惑星アルベド planetTexture: 解像度は detail≥8 で 1536×768、それ以外 512×256。値ノイズ(64 格子、双線形、fbm 4〜5 オクターブ)を基に、rocky はノイズ勾配で base→alt→band を混合、gas は緯度×bands の正弦帯 + 乱流、ice は低周波混合 + かすかな帯。poleCap があれば高緯度を pole 色へ。
- 高さマップ heightTexture: 同じ texSeed から同じ第一ノイズ場を再現した濃淡(detail≥8 で 512×256、それ以外 256×128)。rocky bumpScale 0.03、ice 0.015、gas は bump なし。マテリアルは MeshStandardMaterial(roughness 1, metalness 0)。
- 大気: 裏面・加算のフレネルシェーダ(強度 gas 0.85 / その他 0.55、pow 3)。
- 環 ringTexture: 半径方向 1D(detail≥8 で 1024、それ以外 256)。UV は RingGeometry の頂点半径から張り直す。
- 光点スプライト markerMaterial(colorCss): 64px。ハロ(中心 α0.14 → 0.35 位置 α0.05 → 縁 0)を敷き、その上に鋭い芯(半径 7/64、α1 → 0.85 → 0)。
- 銀河の滲み smudgeTexture(tintCss): 96px の円形(α 0.5 / 0.24 / 0.06 / 0)+ 小さな核(半径 5、α0.95)。3D 平面に貼って傾ける(billboard にしない)。空の隣接銀河のみスプライトで代用し、回転と縦横比で傾きの印象を与える。
- 点テクスチャ pointTexture: 全 THREE.Points に必須の丸いソフトテクスチャ(α 1 → 0.55 → 0)。生の正方形点は禁止。

## 品質と点数の目安 [調整]

品質倍率 mult(std 1 / high 2 / xhigh 4 / still 8)は表示密度と一部テクスチャ解像度のみを変える。実測の目安:

    std   ≈ ダスト 9k + 空 12k + 銀河 2.6k + ミニ 3k        ≈ 27k 点
    high  ≈ 18k + 20.6k + 5.2k + 4.3k + 遠景焼込             ≈ 50k 点
    xhigh ≈ 36k + 35k + 10.4k + 6.1k                          ≈ 90k 点
    still ≈ 80k + 60k + 20.8k + 8.6k + ハロ/星団 2.3k        ≈ 17 万点、pixelRatio 3

点群はクラウドごとに 1 ドローコールのため、負荷の主因は点数ではなく加算スプライトの重なり(フィルレート)である。淡く小さい点を増やす調整は安全側、広いハロやガス層の不透明度を上げる調整は危険側と心得る。
